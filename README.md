# 라이더 관리 플랫폼

## 프로그램 설명

### 프로그램 목표 : 주문이 발생하면 라이더가 주문을 받고, 그에 대한 보상금을 정산받는다

+ 관리자에 의해 `주문`이 발생한다
+ 주문이 발생하면 모든 `유효한 라이더`들에게 `주문번호/주문상세/주문상태`를 전달합니다
+ 라이더들은 `주문번호/주문상세/주문상태`를 `확인`할 수 있습니다
    + 그외의 내용들은 제한됩니다
+ 라이더들은 주문을 `승인/거절`할 수 있습니다
    + 라이더가 주문을 `승인`하면 주문은 해당 라이더에게 `고유하게 할당`됩니다
    + 할당된 주문은 다른 라이더들에게 `거절`을 발생시킵니다
+ 라이더는 배달을 시작하기 위해 주문상태를 `배달중`이라고 `변경`할 수 있습니다
    + 라이더는 자신의 주문이 `배달중`상태일 때 배달완료 이미지를 선택적으로 첨부하여 주문 상태를 `배달완료` 할 수 있습니다
+ 배달완료가 되면 `보상금액`이 `업데이트`됩니다
    + 라이더가 처리한 주문의 갯수에 따라 `보상금액`이 측정됩니다
    + 주문에 대한 `수정/삭제`는 보상금액을 `업데이트`합니다
+ 라이더는 `정산`을 요청할 수 있습니다
+ 관리자가 `승인`을 해야지 `정산`이 완료됩니다
+ 이미 `지급처리한 주문 내역`은 `수정/삭제`가 불가능합니다

``` text
Admin
├─> 라이더 정보 CRUD
├─> 주문정보 CRUD
├─> 보상정보 CRUD
└─> 지급금정보 CRUD

User
├─> 개인라이더 정보 
│   └─>read(내 정보 조회)
├─> 개인주문 정보
│   └─>create(호출 승인->생성)
│   └─>read(내가 현재 받은 주문 조회)
├─> 개인보상 정보
│   └─>create(배달완료 -> 보상 항목 생성)
│   └─>read(현재 보상받을 항목만 보기/정산받은 항목 제외)
└─> 개인지급금 정보 CRUD
    └─>read(지금까지 받은 지급내역 조회)

```

## 도전 사항

+ DDD(Domain Driven Design)를 사용해보자
+ Entity에 대한 필수적인 결합 및 참조
+ 애자일 개발 방식

## 요구사항

### [회원관리]

+ 회원가입 및 로그인
+ 회원은 라이더 < 관리자
+ 일반은 조회만, 관리자는 모든 데이터에 대한 CRUD가 가능

### [라이더]

+ 관리자에 의한 생성/수정/삭제가 가능합니다
+ 라이더를 검색할 수 있습니다
    + 라이더이름, 라이더번호
+ 라이더는 여러 주문을 가지고 있을 수 있습니다(최대 5개)

### [주문 관리]

+ 관리자에 의한 생성/수정/삭제가 가능합니다
+ 주문은 상태를 가집니다
    + 조리중/배달중/배달완료
+ 라이더 하나당 여러 주문이 있고, 하나의 주문은 하나의 주문 상세와 연결되있습니다

### [보상]

+ 관리자에 의한 생성/수정/삭제가 가능합니다
+ 보상은 검색 가능합니다
    + 라이더 이름/라이더 번호
+ 보상 내역을 볼 수 있습니다
    + 라이더에게 종속되있는 보상금액과 주문내역을 볼 수 있습니다

### [지급금 관리]

+ 라이더는 정산을 요청할 수 있습니다
    + 정산 요청 시 지급 날짜를 지정합니다
+ 관리자는 지급금을 승인할 수 있습니다
+ 지급금 상세보기를 할 수 있습니다
    + 정산을 받은 주문 및 보상금액을 볼 수 있습니다
+ 지급금을 검색할 수 있습니다
    + 라이더 번호/라이더이름

### 세부 작업 목록

+ 회원관리 기능 만들기
+ 주문 발생 및 라이더 호출 기능 만들기
+ 보상 정보 기능 만들기
+ 지급금 관리 기능 만들기
